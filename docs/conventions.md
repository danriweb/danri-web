# Конвенции кода (Code Conventions)

В этом документе описаны правила именования, структуры файлов и стиля написания кода в проекте **DanriWeb**, чтобы обеспечить единообразие и поддерживаемость кодовой базы.

## 1. Naming (Именование)

### Папки и директории

- **Формат:** `kebab-case` (в нижнем регистре с дефисами).
- **Примеры:** `language-switcher`, `shadcn`, `mobile-menu`.
- В архитектуре FSD все слои и слайсы (`widgets`, `features`, `entities` и их содержимое) именуются строго в `kebab-case`.

### Компоненты (React)

Правила именования, структуры и написания React компонентов вынесены в [components.md](./components.md).

### Хуки и функции

- **Формат:** `camelCase`.
- **Примеры хуков:** `useMounted.ts`, `useSmoothScroll.ts`.

### Утилиты, сторы, схемы и моки

- **Формат:** `kebab-case` с явным расширением назначения через точку.
- **Сторы:** С суффиксом `.store` (пример: `contact-modal.store.ts`).
- **Схемы:** С суффиксом `.schema` (пример: `contact.schema.ts`).
- **Моки:** С суффиксом `.mock` (пример: `cases.mock.ts`).
- **Утилиты:** В `kebab-case` (пример: `date-formatter.ts`).

## 2. TypeScript и строгая типизация

Проект использует **TypeScript в строгом режиме** (`"strict": true` в `tsconfig.json`).

- Забудьте про `any`. Использование `any` строго запрещено. Заменяйте его на `unknown` или пишите точные интерфейсы.
- Директивы `@ts-ignore` и `@ts-expect-error` допускаются только в самых крайних случаях (когда обойти баг типизации сторонней библиотеки невозможно).
- Для описания контрактов объектов и пропсов компонентов отдавайте предпочтение `interface`, а не `type` (за исключением утилитарных типов).

## 3. Public API в слайсах (FSD)

Подробно о слоях (widgets, features, entities, shared) написано в [architecture.md](./architecture.md). Главное архитектурное правило для работы внутри слоёв:

### Public API (`index.ts`)

Каждый слайс обязан иметь `index.ts`. Импорты из других слайсов осуществляются **только** через этот файл. Глубокие импорты запрещены (контролируется линтером **Steiger**).

## 4. Стилизация кода и форматирование

Проект автоматически форматируется с помощью **Prettier**. Конфигурация находится в `.prettierrc`.

### Ключевые правила

- Ширина строки: до **120 символов**.
- В качестве отступов используются пробелы (2 пробела).
- Для сортировки классов Tailwind используется плагин `prettier-plugin-tailwindcss`.

### Порядок импортов

Обеспечивается плагином `@trivago/prettier-plugin-sort-imports`.  
Импорты автоматически сортируются в таком порядке:

1. Сторонние библиотеки (React, Next, Zustand и т.д.).
2. Слой `@app/*`
3. Слой `@widgets/*`
4. Слой `@features/*`
5. Слой `@entities/*`
6. Слой `@shared/*`
7. Shared-обертки и утилиты (`@dom`, `@styles`, `@shadcn`, `@constructors` и др.)
8. Остальные алиасы (`@/*`)
9. Относительные импорты (`./`, `../`)
